# v0.5.2 — tuple returns and BinOp column fixes (2026-04-28)

Fixes two linetable column attribution bugs that surfaced when compiling
functions that return a tuple of locals, and adds BUILD_TUPLE support.

## What's new

### Parenthesised BinOp right-operand column extension

When a BinOp's right operand is itself a BinOp (i.e. was parenthesised
in source), the outer operation's endCol should include the closing `)`.
The `walkExpr` BinOp general-case path now applies `scanEndCol` when
the right child is a BinOp:

```go
if _, isBinOp := n.Right.(*parser2.BinOp); isBinOp {
    rec = fs.scanEndCol(rec)
}
```

This mirrors the identical check that was already present in
`walkExprSkipFirstLocal`. Without the fix, expressions like `0.74*(r-y)`
had their outer span end at col 17 (end of `y`) rather than col 18
(after the `)`).

### Parenthesised tuple column attribution

For `return (y, i, q)`, the parser sets `Tuple.P` to the position of
the opening `(`, one column before the first element. BUILD_TUPLE's
linetable span now starts at `n.P.Col` when that is less than the
first-element start, and extends `lastEnd` forward past the closing `)`:

```go
if tupleStart := byte(n.P.Col); tupleStart < firstStart {
    firstStart = tupleStart
}
lastEnd = fs.scanEndCol(lastEnd)
```

`scanEndCol` is idempotent when there is no `)` at the scan position,
so non-parenthesised tuples (`return a, b, c`) are unaffected.
