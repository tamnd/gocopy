# v0.6.11 — codegen module-level: init+augmented assignment

Date: 2026-04-29
Spec: notes/Spec/1500/1520_gocopy_v0611_codegen_augassign.md
CPython parity: Python/codegen.c (3.14.0): module-level
`compiler_augassign` for `AugAssign` nodes following an init
`Assign`.

The v0.6.11 release teaches the codegen package the next
module-level shape after the v0.6.10 multi/chained-assign
pair: the modAugAssign sequence `<name> = <initInt>` followed
immediately by `<name> <op>= <augInt>`, optionally followed
by no-op statements.

Both the init and aug values must be non-negative ints; the
target name must match across the two statements and be 1..15
ASCII chars. All 12 in-place binary operators are supported:
`+= -= *= /= //= %= **= &= |= ^= <<= >>=`.

This is the first codegen path to exercise the assembler's
inline-cache emission: `BINARY_OP` carries 5 cache words
(10 bytes) that the IR encoder zero-fills automatically based
on `bytecode.CacheSize[op]`. The PEP 626 line table treats
`BINARY_OP + cache` as one 6-unit entry covering columns
`0..augValEnd`.

The 246/246 oracle stays byte-identical. The parallel-output
gate (`TestCodegenParity`) automatically lights up fixtures
074-088 (15 augmented-assign fixtures spanning every
in-place operator the classifier supports).

## Note on roadmap deviation

Spec 1508 originally allocated v0.6.11 to class definitions.
Adding class definitions now means landing class scope
(`__class__` cell, class-cell injection in nested methods),
`LOAD_BUILD_CLASS`, nested code object emission, and
decorator dispatch — all without a working `if`, function
call, or attribute access. The release would land a feature
whose inner machinery cannot yet exist in codegen.
modAugAssign is the next natural integer-only shape — same
LOAD_x value family as modAssign, plus the inline-cache path.
After v0.6.11 codegen owns every assignment shape with
non-folded integer RHSes that exists in the classifier. Class
definitions slide to whichever sub-release has codegen wide
enough to support them; slots 1521-1522 retain their
allocations (comprehensions, driver/perf).

## Added

- `compiler/codegen/visit_aug_assign.go` — `augAssignModule`
  type plus `classifyAugAssignModule`, `buildAugAssignModule`,
  and the `augOpargFromOp` helper mapping `*ast.AugAssign.Op`
  strings to `bytecode.NbInplace*` enum values. The builder
  emits `RESUME 0`, the init `LOAD_SMALL_INT | LOAD_CONST 0`,
  `STORE_NAME 0`, `LOAD_NAME 0`, the aug `LOAD_SMALL_INT |
  LOAD_CONST 1`, `BINARY_OP <oparg>` (5 cache words appended
  by the IR encoder), `STORE_NAME 0`, `(M-1)` NOPs for the
  no-op tail, then the trailing `LOAD_CONST <noneIdx>;
  RETURN_VALUE` pair. `co.StackSize = 2` (LOAD_NAME +
  augVal both on the stack at BINARY_OP).

## Out of scope

- modAugAssign with negative init or aug values
  (`x = -1; x += 1`) — needs the constant-fold pass.
- modAugAssign with non-int values
  (`x = 0.0; x += 1.5`) — the classifier itself rejects these.
- `x[i] += 1`, `x.f += 1` — different AugAssign target kinds.
- modBinOpAssign, modUnaryAssign, modCmpAssign, modBoolOp,
  modTernary — later sub-releases.
- Function definitions, classes, comprehensions, try/except —
  later.
- Optimizer passes — later.
