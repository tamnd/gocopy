## v0.0.7

`gocopy compile` accepts a leading `name = literal` assignment as the
first statement, where literal is one of `None`, `True`, `False`, or
a plain-ASCII string literal (single-, double-, or triple-quoted, no
escapes, no embedded matching quote). The assignment may be followed
by the same no-op tail v0.0.4..0.0.6 already accepted.

CPython lowers the assignment to:

    RESUME 0
    LOAD_CONST <value>          (const index 0)
    STORE_NAME <name>           (name index 0)
    [no-op tail body]
    LOAD_CONST <None>           (index 0 if value is None, else 1)
    RETURN_VALUE

The `consts` tuple is `(value, None)` when value is non-None and
`(None,)` when value is None itself (CPython collapses the two const
slots in that case). The `names` tuple is `(name,)` — the first time
gocopy emits a non-empty names tuple.

The line table picks up two new pieces:

    prologue: LONG length 1 (covers RESUME)
    LOAD_CONST entry: ONE_LINE? length 1 with the value's column range
    STORE_NAME entry: SHORT0 length (3 if no tail else 1) with the
                      name's column range
    tail: same v0.0.4 rule as no-op bodies

The SHORT0 entry is new: it encodes (start_col, end_col) in a single
payload byte where start_col fits in 3 bits and end_col is at most 15
columns past start_col. CPython uses it for STORE_NAME because the
target name span always fits.

### Added

- `bytecode.AssignBytecode(noneIdx, tailStmts)` for the assignment
  instruction stream (RESUME + LOAD_CONST + STORE_NAME + max(0, t-1)
  NOPs + LOAD_CONST None + RETURN_VALUE).
- `bytecode.AssignLineTable(line, nameLen, valStartCol, valEndCol,
  tail)` for the prologue + ONE_LINE? value entry + SHORT0 store
  entry + no-op tail.
- A `name = literal` parser branch in `compiler.classify` covering
  None, True, False, and ASCII string literals on the right-hand
  side. Identifiers are limited to 1..15 ASCII name chars (the
  SHORT0 entry's end_col-start_col field caps at 15).
- Six new fixtures: `029_assign_none.py` through
  `034_assign_long_name.py`.

### Changed

- `compiler.classify` records a fourth body shape `modAssign`. The
  assignment must be the first statement; later assignments fall
  through to the unsupported path.

### Out of scope (deferred)

- Right-hand side `Ellipsis` (`x = ...`) and bytes literals
  (`x = b"hi"`); both need the marshal layer to learn one new tag
  each (`TYPE_ELLIPSIS`, the bytes path) and ride a separate slot
  rule for bytes consts.
- Right-hand side integer / float / complex literals; `LOAD_SMALL_INT`
  and the `TYPE_INT` / `TYPE_FLOAT` marshal paths are still pending.
- Multi-target assignment (`a = b = None`), augmented assignment
  (`x += 1`), and any expression on the RHS beyond a single literal.
- Assignment after a leading docstring or after another statement;
  v0.0.7 only accepts an assignment as the first statement.
- Wiring gopapy as the parser; still waiting on a gopapy v1.0.0.
