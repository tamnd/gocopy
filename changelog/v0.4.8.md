# v0.4.8 — CPython colorsys._v compilation (2026-04-28)

Extends the compiler to handle a module body of N ≥ 1 constant-folded BinOp
assignments followed by a function definition, and fixes the LFLBLFLB
superinstruction to cover non-trivial right-hand sub-expressions.

## What's new

### Generalised LFLBLFLB (LOAD_FAST_BORROW_LOAD_FAST_BORROW)

Previously gocopy only applied the LFLBLFLB superinstruction when both children
of an outer BinOp were direct Name nodes. CPython applies it whenever the left
child is a local Name and the first load of the right child is also a local Name,
regardless of how deeply nested the right-hand expression is.

Two new helpers on `funcState`:
- `peekFirstLocalSlot(e)` — follows the leftmost AST path to find the first
  local slot that would be loaded, returning `(0, false)` for non-local or
  non-BinOp paths.
- `walkExprSkipFirstLocal(e)` — emits bytecode for `e` assuming its first
  LOAD_FAST_BORROW was already consumed by a preceding LFLBLFLB. For BinOp
  nodes it recurses into the left child via `walkExprSkipFirstLocal` and walks
  the right child via the normal `walkExpr`.

### Parenthesis column tracking in `walkExprSkipFirstLocal`

CPython attributes the opening `(` of a parenthesised sub-expression to the
outer operation that consumes it as its left operand, and the closing `)` to the
outer operation that consumes a parenthesised right operand. Two adjustments are
applied inside `walkExprSkipFirstLocal(BinOp)`:
- When `n.Left` is a BinOp: scan backward one column with `scanBackOpen`; if
  the preceding character is `(`, include it in the BINARY_OP startCol.
- When `n.Right` is a BinOp: call `scanEndCol` after walking the right side;
  if the next character is `)`, include it in the BINARY_OP endCol.

A new `scanBackOpen` helper checks `srcLines[stmtLine-1][col-1]` and returns
`col-1` if it is `(`.

### New classification: `modAssignsThenFuncDef`

A module body of N ≥ 1 assignments whose values are all `foldedBinOp` constants
followed by exactly one `funcBodyExpr` function definition.

co_consts layout: `(leftVal_of_first_fold, innerCode, None, result_fold_1,
result_fold_2, …)` — the first fold's left operand precedes the inner code
object so LOAD_CONST indices stay compact; subsequent folds' results are
appended in declaration order.

Bytecode per assignment: `LOAD_CONST foldIdx + STORE_NAME nameIdx`. Then the
funcdef: `LOAD_CONST codeObjIdx + MAKE_FUNCTION + STORE_NAME funcNameIdx`.
Tail: `LOAD_CONST noneIdx + RETURN_VALUE`.

### `compileFuncBodyCore` helper

Extracts the inner `CodeObject` from the outer module wrapper produced by
`compileFuncBodyExpr`, then decodes `bodyEndLine` and `bodyEndCol` from the
outer module's linetable. Used by `compileAssignsThenFuncDef` to build the
combined module linetable without duplicating the funcdef logic.

### `AssignsThenFuncDefLineTable` in `bytecode/funcdef.go`

Generates the PEP 626 linetable for a module body of N assignments followed by
a funcdef: RESUME prologue, then one `appendValueEntry` + `appendShort0Entry`
pair per assignment, then a LONG entry covering the 5 funcdef CUs spanning
`defLine..bodyEndLine`.
