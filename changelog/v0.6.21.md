# v0.6.21 — codegen module-level: genExpr (closes band A)

Date: 2026-04-29
Spec: notes/Spec/1500/1531_gocopy_v0621_codegen_gen_expr.md
Roadmap: notes/Spec/1500/1523_gocopy_v06x_part2_roadmap.md (band A)
CPython parity: Python/codegen.c (3.14.0): module-level
`compiler_visit_expr` recursion driven by a top-level
`Assign` whose value is a non-folded compound expression.

The v0.6.21 release is the eighth and final sub-release in
band A of the v0.6.x part 2 roadmap. It teaches the codegen
package the modGenExpr shape:

- `<target> = <expr>` where target is a Name of 1..15 ASCII
  chars and `<expr>` is recursively composed of Name,
  small-int Constant (0..255), BinOp (any of the 13
  supported operators), or UnaryOp (USub or Invert), all on
  a single source line. Multi-statement modules fall
  through (the classifier path keeps single-statement
  parity for tail no-ops too — both produce identical
  bytecode).

This is the first codegen path that is itself a recursive
walker rather than a closed-form template, and the first to
emit `LOAD_SMALL_INT`, `UNARY_NEGATIVE`, and
`UNARY_INVERT`. It generalises the BinOp / UnaryOp patterns
modBinOpAssign and modUnaryAssign already established in
v0.6.12 and v0.6.13 — those slots still own their dedicated
fixtures because Build dispatches modGenExpr last.

Notable byte-equality details, all mirroring the
classifier's `compileGenExpr` byte-for-byte:

- **co_names**: insertion order, first occurrence wins.
- **co_consts**: only the first int constant encountered
  during the walk is appended before the trailing None.
  Subsequent ints reach `LOAD_SMALL_INT` directly via oparg
  and never enter co_consts. CPython's compiler tracks this
  same way.
- **BinOp Loc**: spans `(leftStartCol, rightEndCol)` —
  encompasses the whole subtree.
- **UnaryOp Loc**: spans `(opCol, operandEndCol)` —
  starts at the unary operator, ends where the operand
  ends.
- **StackSize**: recursive `max(leftDepth, rightDepth+1)`,
  with a floor of 2 to match the assembler's stack-depth
  analysis on the simple Names-only fixtures.
- **Folded shapes rejected**: bare Constants,
  UnaryOp(USub, numeric Constant), and BinOp(numeric
  Constant, op, numeric Constant) all fall through so the
  existing modAssign path keeps owning negative literals
  and folded constant BinOps.

The 246/246 oracle stays byte-identical. The parallel-
output gate (`TestCodegenParity`) automatically lights up
fixtures 138-143:

- 138 `nested_binop_aaa.py`  (`x = a + b + c`)
- 139 `nested_binop_abcd.py` (`x = a * b + c * d`)
- 140 `nested_binop_prec.py` (`x = a + b * c`)
- 141 `binop_lit_right.py`   (`x = a + 1`)
- 142 `binop_lit_left.py`    (`x = 1 + a`)
- 143 `unary_in_binop.py`    (`x = -a + b`)

Band A is now complete. Every module-level classifier shape
that fits the assignment-with-immediate-right-hand-side
family lives in codegen. Bands B (function definitions,
classes, comprehensions, try/except), C (optimizer passes,
classifier funeral) and D (perf) follow from v0.6.22
onward.

## Added

- `compiler/codegen/visit_gen_expr.go` — `genExprModule`
  type plus `classifyGenExprModule` + `buildGenExprModule`
  with a recursive `genWalker` that mirrors
  `compiler/gen_expr.go::genState` byte-for-byte.
  Single-statement only so the parity gate stays byte-equal
  with the classifier's `compileGenExpr`.

## Changed

- `compiler/codegen/codegen.go` — `Build` recognizes
  modGenExpr after modCallAssign, ahead of the
  ErrUnsupported fallback. Doc comment updated to record
  band A completion.

- `compiler/codegen_parity_test.go` — `compileViaClassifier`
  picks up the modGenExpr classifier path so the parity
  gate validates the new codegen-owned fixtures against the
  classifier reference.

## Tests

- 246/246 fixtures byte-identical (`tests/run.sh`).
- `compiler/codegen/codegen_test.go` adds:
  `TestBuildGenExprAddChain`,
  `TestBuildGenExprNestedPrec`,
  `TestBuildGenExprIntRight`,
  `TestBuildGenExprUnaryThenAdd`,
  `TestBuildGenExprWithTailFallsThrough`.
- `compiler/codegen_parity_test.go` automatically validates
  fixtures 138, 139, 140, 141, 142, and 143 via the new arm.

## Out of scope

- Float / complex / string constants — small-int only.
- BoolOp / Compare / IfExp inside a genExpr — those land in
  modBoolOp / modCmpAssign / modTernary which already own
  dedicated fixtures.
- Call / Attribute / Subscript inside a genExpr.
- Function bodies — band B starts in v0.6.22.
- Optimizer passes, classifier funeral, perf — bands C and
  D.
