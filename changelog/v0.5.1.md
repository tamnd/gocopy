# v0.5.1 — multi-funcdef modules (2026-04-28)

Adds support for a module body consisting entirely of two or more function
definitions with no other top-level statements.

## What's new

### New classification: `modMultiFuncDef`

`stmtsToClassification` now distinguishes three cases when the first statement
is a `stmtFuncBodyExpr`:

1. All remaining statements are noops — existing `modFuncBodyExpr` path.
2. All statements (including the first) are `stmtFuncBodyExpr` — new
   `modMultiFuncDef` path.
3. Anything else — unsupported.

The classification carries `multiFuncDefAsgns []funcBodyInfo`, one entry per
function in source order.

### `compileMultiFuncDef`

Compiles each function body by delegating to the existing `compileFuncBodyCore`
helper, then assembles the module code object:

- **co_consts**: `[code_0, code_1, …, code_{n-1}, None]`
- **co_names**: function names in source order
- **Bytecode**: `RESUME` then N×(`LOAD_CONST i` + `MAKE_FUNCTION` +
  `STORE_NAME i`), then `LOAD_CONST n` + `RETURN_VALUE`

### `MultiFuncDefLineTable` in `bytecode/funcdef.go`

Generates the PEP 626 linetable for N function definitions:

- Prologue: `f0 03 01 01 01` (RESUME, 1 CU, no-source)
- For each funcdef except the last: LONG entry covering 3 CUs
  (LOAD_CONST + MAKE_FUNCTION + STORE_NAME)
- For the last funcdef: LONG entry covering 5 CUs
  (LOAD_CONST + MAKE_FUNCTION + STORE_NAME + LOAD_CONST None + RETURN_VALUE)

### Marshal FLAG_REF fix for `co_localspluskinds`

CPython creates a fresh `bytes` object for `co_localspluskinds` per code object
even when the content is identical across functions. Each object has REFCNT=1
so CPython's marshaller never sets FLAG_REF on it. gocopy's ref-counter now
handles this via a dedicated `localspluskinds` method that only tracks the
empty-bytes singleton (which CPython interns globally, so it does get
FLAG_REF). Non-empty localspluskinds bytes are not counted globally, ensuring
the flag is never set on them.

This is distinct from `co_linetable`, where CPython does share the same `bytes`
object between functions whose linetable content happens to be identical
(REFCNT > 1), so FLAG_REF is correctly set there.
