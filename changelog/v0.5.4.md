# v0.5.4 — if-elif-else assign chains with JUMP_FORWARD (2026-04-28)

Adds support for if/elif/else chains where every branch assigns to the same
variable (or introduces it) in function bodies. This is the pattern CPython
uses inside `rgb_to_hsv` to compute the hue component.

## What's new

### isIfElseAssign: new function body statement type

`extractIfElseAssignChain` walks an if/elif/else AST node and accepts it when:
- Every branch body is a single `Assign` with the same target name.
- The else branch is present (so every path assigns the variable).
- The target need not be in `knownNames` yet; if it is new it gets allocated
  a fresh FAST slot.

The emitter in `func_body.go` generates, for each non-else branch:

```
[condition]   POP_JUMP_IF_FALSE → next_branch
NOT_TAKEN
[body expr]   STORE_FAST(target)   JUMP_FORWARD → end
```

and for the else branch:

```
[body expr]   STORE_FAST(target)
```

All PJIF and JUMP_FORWARD targets are backpatched after the entire chain is emitted.

### STORE_FAST + JUMP_FORWARD share one linetable entry

CPython combines STORE_FAST and the following JUMP_FORWARD into a single
same-line linetable entry (2 + cache_words code units). The emitter follows
suit: for non-else branches, a single `emitSame(1 + 1 + jfCacheWords, tsc, tec)`
covers both instructions.

## Fixtures added

- `235_if_else_assign.py` — `pick(r, g, b)`: `max()` call, 3-branch
  if/elif/else assigning `h`, then `h = h / 6.0` to force JUMP_FORWARD
  (without the trailing computation CPython would inline the return into
  each branch, eliminating JUMP_FORWARD entirely).
