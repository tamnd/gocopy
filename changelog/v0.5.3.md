# v0.5.3 — CALL with LFLBLFLB args, tuple and BinOp column fixes (2026-04-28)

Three more linetable and bytecode fixes, all surfaced by trying to compile
a function that calls max/min with three local arguments and returns an
unparenthesised tuple.

## What's new

### CALL argument loading: LFLBLFLB for N ≥ 2 args

The previous CALL implementation applied LFLBLFLB only when `nArgs == 2`.
For `max(r, g, b)`, this emitted three individual LOAD_FAST_BORROW
instructions instead of LFLBLFLB(r, g) + LOAD_FAST_BORROW(b).

The condition now reads `nArgs >= 2`: when the first two call arguments are
both local Names with slots 0-15, emit LFLBLFLB for them and load the
remaining arguments individually.

### Tuple element loading: LFLBLFLB at any consecutive local pair

Previously LFLBLFLB was only attempted for elements [0] and [1]. CPython
applies the optimisation at any consecutive pair of local Names within the
element list.

For `return 0.0, s, v` (slots s=6, v=5): element 0 is a float constant, so
[0,1] doesn't qualify. But [1,2] = (s, v) are both locals → LFLBLFLB(s, v).

The single-pass loop now checks LFLBLFLB at every position before falling
back to `walkExpr`.

### BinOp general case: scanBackOpen for parenthesised left operand

When a BinOp's left child is itself a BinOp (i.e. was parenthesised in
source), CPython's span for the outer operation starts at the `(`. The
general-case path now calls `scanBackOpen(lsc)` when the left child is a
BinOp, matching the identical check that was already in `walkExprSkipFirstLocal`.

Without the fix, `(maxc-minc) / maxc` had outer `/` span starting at `m`
(col 9) instead of `(` (col 8).
