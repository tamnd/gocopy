## v0.0.4

`gocopy compile` accepts blank or comment-only lines anywhere in the
source: leading, trailing, or between no-op statements. The
v0.0.3 grammar required statements on consecutive lines starting at
line 1; this rung lifts that constraint without enlarging the
no-op token set.

What CPython does once a statement is no longer on `prev_line + 1`:

- delta = 0 (multi-statement on one line, still out of scope) -
  ONE_LINE0 entry.
- delta = 1 - ONE_LINE1 (already covered in v0.0.3).
- delta = 2 - ONE_LINE2.
- delta >= 3 - LONG entry with svarint(line_delta), end_line_delta=0,
  varint(start_col+1), varint(end_col+1).

A leading blank pushes the first statement off line 1, which
collapses to a single LONG entry covering the whole body.

We mirror all of that exactly. Verified against `python3.14 -m
py_compile` for gaps from one line up through ten blank lines.

### Added

- `bytecode.NoOpStmt{Line, EndCol}` carries the source line of each
  statement so the encoder can compute deltas.
- Private `bytecode.appendVarint` and `bytecode.appendSignedVarint`
  primitives implementing CPython's base-64 varint and zigzag
  svarint from `Objects/locations.md`.
- Five new fixtures: `014_pass_blank_pass.py` through
  `018_mixed_gaps.py` covering one-line gap, multi-line gap,
  leading blanks, comment-only gap, and three statements with
  mixed deltas.

### Changed

- `bytecode.LineTableNoOps` now takes `[]NoOpStmt` instead of
  `[]byte`. The single-no-op helper still exists as a one-line
  wrapper.
- The classifier accepts blank/comment lines anywhere and records
  each statement's source line.

### Out of scope (deferred)

- Multiple statements on the same line (`pass; pass`). The encoder
  could already emit ONE_LINE0 but the scanner has no semicolon
  parser; this whole branch waits on real parsing.
- String / bytes literal as a top-level statement (docstring path).
- gopapy as the parser (waits on a gopapy v1.0.0 cut; see v0.0.2
  notes).
