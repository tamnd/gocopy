# v0.7.10.3 — insert_superinstructions 1:1 port

Date: 2026-04-30
Spec: notes/Spec/1500/1561_gocopy_v07103_insert_superinstructions.md
Roadmap: notes/Spec/1500/1558_gocopy_v0710x_release_roadmap.md

v0.7.10.3 ports CPython 3.14
`Python/flowgraph.c::insert_superinstructions` into the gocopy
flowgraph package as the first flowgraph pass. The visitor
stops pre-fusing `LOAD_FAST_BORROW` pairs at emit time;
the new pass owns LFLBLFLB fusion (and the three
official CPython super-instruction pairs).

## Added

- `compiler/flowgraph/insert_superinstructions.go` —
  - `InsertSuperinstructions` (mirrors
    Python/flowgraph.c:2588 `insert_superinstructions`).
  - `makeSuperInstruction` (mirrors Python/flowgraph.c:2572
    `make_super_instruction`).
- `compiler/flowgraph/insert_superinstructions_test.go` — 8
  unit tests exercising every supported pair, both fuse-
  rejection guards (slot ≥ 16, line mismatch), the gocopy
  borrow-pair bridge, and the nil-seq edge case.
- `bytecode/opcode.go` — three CPython super-instruction
  opcodes that were not yet in gocopy:
  - `LOAD_FAST_LOAD_FAST` (89)
  - `STORE_FAST_LOAD_FAST` (113)
  - `STORE_FAST_STORE_FAST` (114)
- `bytecode/opmeta.go` — opmeta entries for the three new
  super-instruction opcodes.

## Changed

- `compiler/optimize/optimize.go::Run` — pipeline now runs
  `flowgraph.InsertSuperinstructions(seq)` between
  `inlineSmallExitBlocks` and `resolveJumps`, approximating
  CPython's `_PyCfg_OptimizeCodeUnit` ordering
  (Python/flowgraph.c:3659).

## Removed

- `compiler/codegen/visit_func_stmt.go::fuseLflblflbTail` —
  the visitor's emit-time LFLBLFLB pre-fuser. Its
  responsibility moves to the flowgraph pass.
- The corresponding `fuseLflblflbTail(u)` callsite in
  `visitFuncNameExpr`.

`visit_func_stmt.go` shrinks by ~35 LOC.

## Deviation from CPython

CPython's `insert_superinstructions` only fuses
`LOAD_FAST/STORE_FAST` pairs (the borrow promotion happens
later, in `optimize_load_fast`). The gocopy visitor still
emits `LOAD_FAST_BORROW` directly at v0.7.10.3, so the pass
also fuses

```
LOAD_FAST_BORROW + LOAD_FAST_BORROW
  → LOAD_FAST_BORROW_LOAD_FAST_BORROW
```

This bridge is removed in v0.7.10.4 once the visitor emits
raw `LOAD_FAST` and `optimize_load_fast` owns the borrow
promotion. The deviation is documented in the file header.
