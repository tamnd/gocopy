# v0.5.5 — scanBackOpen guard, tuple depth fix, colorsys YIQ/HLS (2026-04-28)

Three correctness fixes and four new CPython colorsys functions as fixtures.

## What's new

### scanBackOpen: guard against paren wrapping the entire expression

`scanBackOpen` scans back from the left child's column to find a `(`. Previously
it would fire whenever the left child was a BinOp and the char immediately before
it was `(` — including cases where that `(` wraps the ENTIRE right-hand expression,
not just the left sub-expression.

Example: in `rangec / (2.0-maxc-minc)`, the right child `(2.0-maxc-minc)` is
a parenthesized BinOp. Its LEFT sub-expression is `2.0-maxc`, and the `(` at
col 21 wraps the whole thing. When walking the inner `(2.0-maxc)-minc`, the
old code incorrectly scanned back and included the `(` at col 21 as if it were
a paren around `2.0-maxc` alone.

Fix: after `scanBackOpen` finds a `(` at column X, call `scanMatchingClose(X)`
to find the matching `)`. Only apply `scanBackOpen` if the `)` closes before the
right child's start column. Applied to both the general BinOp path in `walkExpr`
and to `walkExprSkipFirstLocal`.

### Tuple element depth: account for previously-loaded elements

When emitting a tuple like `(e0, e1, e2)`, each element `ei` is evaluated with
`i` prior results already on the stack. The `trackDepth` calls inside `walkExpr`
did not account for this base depth, causing `co_stacksize` to be underestimated
for tuples whose elements contain deep sub-expressions (e.g. calls with 3+ args).

Fix: save/restore `fs.maxDepth` around each `fs.walkExpr(n.Elts[i])` call and
merge the element's peak depth as `i + elemPeak`.

This fixes `hls_to_rgb`, whose return expression
`(_v(m1,m2,h+ONE_THIRD), _v(m1,m2,h), _v(m1,m2,h-ONE_THIRD))` reaches a peak
stack depth of 8 (not 6 as previously computed).

### New helper: scanMatchingClose

Added `scanMatchingClose(openCol)` to find the `)` that matches a `(` at a given
column on the current source line. Traverses the line maintaining paren depth.
