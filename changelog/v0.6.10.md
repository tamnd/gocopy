# v0.6.10 — codegen module-level: multi/chained assignment

Date: 2026-04-29
Spec: notes/Spec/1500/1519_gocopy_v0610_codegen_multi_chain_assign.md
CPython parity: Python/codegen.c (3.14.0): module-level
`compiler_mod` → `compiler_visit_stmt` for `Assign` nodes
with N >= 1 targets and a non-folded simple-constant RHS.

The v0.6.10 release teaches the codegen package the next two
module-level shapes after the v0.6.9 single-target assign:

- **modMultiAssign**: `N >= 2` independent `<name> = <const>`
  statements in sequence, optionally followed by no-op
  statements.
- **modChainedAssign**: a single `t0 = t1 = ... = tN-1 =
  <const>` statement with `N >= 2` Name targets, optionally
  followed by no-op statements.

Values stay restricted to the v0.6.9 simple-constant set (None,
True, False, int, float, complex, str, bytes, Ellipsis); names
remain 1..15 ASCII chars. The classifier still owns the
constant-folded sub-shapes (`x = -1`, `x = y = -1`, `x = 1 +
2`), tuple unpacking (`x, y = 1, 2`), augmented assignment
(`x += 1`), and every other module-level shape.

The 246/246 oracle stays byte-identical. The parallel-output
gate (`TestCodegenParity`) automatically lights up every
fixture matching the new subsets — including 061-065 (multi-
assign), 068-071, 073 (chained-assign).

## Note on roadmap deviation

Spec 1508 originally allocated v0.6.10 to try/except/finally +
the PEP 657 exception-table encoder. With codegen still owning
only empty/no-ops/docstring/single-assign after v0.6.9, adding
exception handling now would land a feature that nothing else
in codegen can yet stand alongside — no function calls, no
control-flow `if`, no multi-target assignment. Try/except
slides to whichever sub-release has codegen wide enough to
support it; spec slots 1520-1522 retain their roadmap
allocations (classes, comprehensions, driver/perf).

## Added

- `compiler/codegen/visit_multi_assign.go` — `multiAssignModule`
  type plus `classifyMultiAssignModule`, `buildMultiAssignModule`,
  and the per-statement `classifyAssignOne` helper. The builder
  emits `RESUME 0`, then for each assignment a `LOAD_SMALL_INT`
  (int64 in 0..255) or `LOAD_CONST` followed by `STORE_NAME`,
  then `(M-1)` NOPs for the no-op tail, then the standard
  `LOAD_CONST <noneIdx>; RETURN_VALUE` trailing pair sharing
  either the last assignment's name Loc (no tail) or the last
  tail stmt's Loc.

- `compiler/codegen/visit_chained_assign.go` — `chainedAssignModule`
  type plus `classifyChainedAssignModule` and
  `buildChainedAssignModule`. The builder emits `RESUME 0;
  LOAD_x <val>`, then for each target except the last `COPY 1;
  STORE_NAME`, then a final `STORE_NAME`, the no-op tail, and
  the trailing return pair. `co.StackSize = 2` (COPY pushes a
  second item).

## Changed

- `compiler/codegen/codegen.go` — `Build` recognizes the
  modMultiAssign shape after modAssign and the
  modChainedAssign shape after modMultiAssign, ahead of the
  ErrUnsupported fallback.

- `compiler/codegen_parity_test.go` — `compileViaClassifier`
  picks up the modMultiAssign and modChainedAssign classifier
  paths so the parity gate validates every newly-codegen-owned
  fixture against the classifier reference. The arms delegate
  to the classifier's `compileMultiAssign` /
  `compileChainedAssign` to mirror the exact value-dispatch
  rules (small-int phantom slot, None collapse, deferred
  negLiteral/foldedBinOp slots).

- `compiler/codegen/codegen_test.go` — replaced
  `TestBuildChainedAssignReturnsErrUnsupported` (the shape it
  asserted now goes through codegen) with
  `TestBuildTupleAssignFallsThrough` covering tuple unpacking,
  which remains classifier territory.

## Out of scope

- modMultiAssign / modChainedAssign with `negLiteral` (`x =
  -1`) — needs the constant-fold pass.
- modMultiAssign / modChainedAssign with `foldedBinOp` (`x =
  1 + 2`) — needs the constant-fold pass.
- Tuple unpacking (`x, y = 1, 2`) — different AST shape; future
  release.
- modAugAssign, modBinOpAssign, modUnaryAssign, modCmpAssign,
  modBoolOp, modTernary — later sub-releases.
- Function definitions, classes, comprehensions, try/except —
  later.
- Optimizer passes — later.
