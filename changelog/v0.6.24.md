# v0.6.24 — codegen module-level: for loop (closes band B)

Date: 2026-04-29
Spec: notes/Spec/1500/1534_gocopy_v0624_codegen_for.md
Roadmap: notes/Spec/1500/1523_gocopy_v06x_part2_roadmap.md (band B)
CPython parity: Python/codegen.c (3.14.0): module-level
`compiler_for` lowering of a `For` whose body is a single
small-int assignment.

The v0.6.24 release closes band B of the v0.6.x part 2
roadmap. It teaches the codegen package the modFor shape:

- A single top-level `for loopVar in iter: bodyVar = val`
  loop with no break/continue/else, where iter and loopVar
  are 1..15-char ASCII Names and val is a small int (0..255).

This is the third backward-jump-bearing release after
modWhile. The bytecode pattern uses the FOR_ITER + END_FOR +
POP_ITER opcode trio (FOR_ITER carries 1 inline-cache word).
As with modIfElse and modWhile, the jump opargs are constant
for this exact shape (FOR_ITER 5, JUMP_BACKWARD 7), so codegen
still emits one straight-line block of IR with hand-coded
opargs.

Notable byte-equality details, all mirroring the classifier's
`compileFor` byte-for-byte:

- **Setup run** (4 cu): `LOAD_NAME iterIdx` / `GET_ITER 0` /
  `FOR_ITER 5` (+1 cache) — all attributed to the iterator's
  source position on the for-line.
- **Loop top**: `STORE_NAME loopVarIdx` (1 cu) at the loop
  variable's source position on the for-line.
- **Body**: `LOAD_SMALL_INT bodyVal` (1 cu) at the value's
  body-line position, then `STORE_NAME bodyVarIdx` /
  `JUMP_BACKWARD 7` (+1 cache) — 3-cu run at the body
  variable's source position.
- **Loop exit** (4 cu): `END_FOR 0` / `POP_ITER 0` /
  `LOAD_CONST noneIdx` / `RETURN_VALUE 0` — all attributed
  back to the iterator's source position via a LONG line-
  table entry (negative `lineDelta` from bodyLine to
  forLine), byte-equal with the classifier's hand-written
  LONG entry in `bytecode.ForAssignLineTable`.
- **co_consts**: `[int64(BodyVal), nil]`.
- **co_names**: insertion-order, deduped — iterName,
  loopVarName, bodyVarName.
- **StackSize**: 2 (the iterator stays on the stack while
  the body runs).

The 246/246 oracle stays byte-identical. The parallel-output
gate (`TestCodegenParity`) automatically lights up fixture
135 `for_simple.py`.

Band B is now complete. After v0.6.24 every classifier shape
that survives without nested code objects lives in codegen.
Band C (v0.6.25 onward) introduces function code objects with
locals, arguments, and the cell/free closure machinery —
modFuncDef opens it.

## Added

- `compiler/codegen/visit_for.go` — `forModule` type plus
  `classifyForModule` + `buildForModule`. Mirrors
  `compiler/compiler.go::compileFor` and
  `compiler/classify_ast.go::extractForAssign` byte-for-byte.
  Single-statement only so the parity gate stays byte-equal
  with the classifier.

## Changed

- `compiler/codegen/codegen.go` — `Build` recognizes modFor
  after modWhile, ahead of the ErrUnsupported fallback. Doc
  comment updated to record band B closing and the new
  FOR_ITER + END_FOR + POP_ITER opcode trio.

- `compiler/codegen_parity_test.go` — `compileViaClassifier`
  picks up the modFor classifier path so the parity gate
  validates the new codegen-owned fixture against the
  classifier reference.

## Tests

  `TestBuildForSimple`,
  `TestBuildForSameNamesDedupe`,
  `TestBuildForMultiVarFallsThrough`,
  `TestBuildForWithElseFallsThrough`,
  `TestBuildForNonNameIterFallsThrough`.
- `compiler/codegen_parity_test.go` automatically validates
  fixture 135 via the new arm.

## Out of scope

- `break` / `continue` / `else:` clauses.
- Multi-statement bodies, non-int values, nested loops.
- Compound iterators (Call / Subscript / BinOp).
- Function-body for loops — band C territory.
- Optimizer passes, classifier funeral, perf — bands C and D.
