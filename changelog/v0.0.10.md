## v0.0.10

`gocopy compile` accepts a float literal on the right-hand side of a
leading `name = literal` assignment. Any value `strconv.ParseFloat`
accepts is valid тАФ `1.0`, `3.14`, `0.0`, `1e100`, `1_000.5` тАФ except
complex literals (trailing `j`/`J` are rejected).

CPython always uses `LOAD_CONST 0` for floats (no `LOAD_SMALL_INT`
path). `consts = (float_val, None)`, `noneIdx = 1`. The no-op tail
and line-table shape are identical to the v0.0.7 string/None assign.

Marshal emits `TYPE_BINARY_FLOAT` (0x67): 8 bytes, IEEE 754
double-precision, little-endian. Floats are not immortal in CPython
3.14, so `FLAG_REF` is set only when the same float value appears more
than once in the const-walk (which does not happen in the plain
assignment case, so the byte is always `0x67`).

### Added

- `marshal.emitObject` handles `float64` тЖТ `TYPE_BINARY_FLOAT`
  (8-byte little-endian double). No `FLAG_REF` unless the value
  appears more than once in the walk.
- `float64KeyType` and `float64Key()` in `marshal/writer.go`: key by
  bit pattern so `NaN != NaN` compares correctly.
- `tupleKey` emits `'G'` (8 bytes) for `float64` items.
- `refCounter.tuple` bumps `float64Key(x)` for `float64` items.
- `parseFloatLiteral` in `compiler/classify.go`: accepts any
  `strconv.ParseFloat`-valid token containing `.`, `e`, or `E`;
  rejects complex (`j`/`J` suffix). Underscore separators stripped
  before parsing.
- `tryParseAssign` tries `parseFloatLiteral` between the integer and
  string/bytes branches.
- Five new fixtures: `048_assign_float_one.py` through
  `052_assign_float_then_pass.py`.

### Out of scope (deferred)

- Negative float literals (unary minus not yet parsed).
- Float values in the no-op statement set (bare `1.0` as a no-op is
  already accepted via the v0.0.2 classifier; this rung only adds the
  assignment path).
- Docstring + assignment combo.
