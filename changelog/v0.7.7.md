# v0.7.7 — visit_While + JUMP_BACKWARD support in resolveJumps

Date: 2026-04-29
Spec: notes/Spec/1500/1549_gocopy_v077_visit_while.md
Architecture: notes/Spec/1500/1541_gocopy_v07_architecture.md
Roadmap: notes/Spec/1500/1542_gocopy_v07x_roadmap.md
CPython parity: Python/codegen.c (3.14.0) `codegen_visit_stmt_while`
(the while-loop statement visitor, ported as `visitWhileStmt`),
`codegen_jump_if` (Name-cond branch, reused verbatim from v0.7.6
as `emitJumpIfFalse`); Python/flowgraph.c (3.14.0)
`resolve_unconditional_jumps` and the relocation half of
Python/assemble.c (3.14.0) `_PyAssemble_ResolveJumps` (the
backward-jump distance computation that lifts the v0.7.4
tripwire).

The seventh v0.7.x feature release. v0.7.6 brought the visitor's
first multi-block CFG (forward-only); v0.7.7 adds the first
backward jump — `JUMP_BACKWARD` in the loop back-edge — which
lifts the `resolveJumps` panic placed at v0.7.4 as the explicit
"this case isn't wired yet" tripwire. The v0.6.23 modWhile
codegen classifier file and Build dispatch arm are gone; the
legacy `compiler/compiler.go::compileWhile` fallback stays alive
for shapes the visitor still rejects (else clause, multi-stmt
body, non-Name cond, non-small-int RHS) until later phase B
widens the visitor surface.

The original v0.7.7 row in the roadmap bundled `visit_While` and
`visit_For` together. v0.7.7 ships visit_While only; visit_For
lands at v0.7.8. modFor stays on the legacy classifier for one
more release.

The 246/246 oracle stays byte-identical and the visitor's parity
count climbs from 138/246 to 139/246 (134_while_simple.py).

## Added

- `compiler/codegen/visit_while_stmt.go` — `visitWhileStmt`, the
  while-loop statement visitor. Allocates two labels: `topLabel`
  (loop top, target of `JUMP_BACKWARD`) and `exitLabel` (loop
  exit, target of `POP_JUMP_IF_FALSE`). Adds a labelled topBlock
  for the cond head, an unlabelled body block carrying NOT_TAKEN
  + the small-int Assign + JUMP_BACKWARD, and a labelled exit
  block carrying the trailing LOAD_CONST None + RETURN_VALUE pair
  at condLoc. Reuses `emitJumpIfFalse` and `emitIfBranchAssign`
  from v0.7.6 verbatim; the loop-exit tail is open-coded into the
  exit block (no `emitImplicitReturnNone` call) to make the
  condLoc choice obvious at the call site. Sets
  `u.tailEmitted = true` so visitModule skips its own trailing
  terminator emit.
- `compiler/optimize/optimize_test.go::TestRunResolvesJumpBackward`
  — asserts that JUMP_BACKWARD's Arg is rewritten to
  `(jumpEnd - target)`, the positive backward distance the
  encoder consumes.
- `compiler/optimize/optimize_test.go::TestRunPanicsOnForwardJumpBackwardTarget`
  — asserts the matching invariant: a forward-jump opcode whose
  target lands before its jumpEnd is a programmer error in the
  visitor or CFG construction code, not a valid CFG.

## Changed

- `compiler/codegen/visit_stmt.go::visitStmt` — adds an
  `*ast.While` arm that dispatches to `visitWhileStmt`.
  `isLast=false` returns `ErrNotImplemented` (the v0.7.7 surface
  covers only top-level single-stmt-body while loops with no
  trailing stmts).
- `compiler/optimize/resolve_jumps.go::resolveJumps` — replaces
  the v0.7.4 backward-jump panic with a per-opcode rewrite:
  `JUMP_BACKWARD` gets `(jumpEnd - target)`, every other jump
  opcode gets `(target - jumpEnd)`. Two narrow invariant panics
  remain: `JUMP_BACKWARD` with non-backward target, and any
  forward jump opcode with backward target. Both encode bugs
  upstream of the optimizer, not valid CFGs.
- `compiler/codegen/codegen.go::Build` — drops the
  `classifyWhileModule` / `buildWhileModule` dispatch arm. `Build`
  now starts at `classifyForModule`; only three statement-level
  shapes (modFor, modFuncDef, modClosureDef) remain on the
  codegen classifier surface. Package doc updated to reflect
  v0.7.7 promotion.

## Removed

- `compiler/codegen/visit_while.go` — the v0.6.23 modWhile
  classifier file (~180 LOC: `whileModule`, `classifyWhileModule`,
  `buildWhileModule`).
- `compiler/codegen/codegen_test.go` — the `TestBuildWhileSimple`,
  `TestBuildWhileSameNameDedupes`,
  `TestBuildWhileMultiVarFallsThrough`,
  `TestBuildWhileWithElseFallsThrough`, and
  `TestBuildWhileNonNameCondFallsThrough` tests that called
  `Build` directly to assert specific bytecode / fall-through for
  the deleted classifier shape. Oracle parity, visitor parity,
  and `TestCodegenParity` still cover the build-side behaviour
  end to end.
- `compiler/optimize/optimize_test.go::TestRunRejectsBackwardJump`
  — replaced by `TestRunResolvesJumpBackward` and
  `TestRunPanicsOnForwardJumpBackwardTarget`. The v0.7.4 tripwire
  asserted "JUMP_BACKWARD always panics"; v0.7.7 splits that into
  the resolution check and the narrower invariant panics.

## Out of scope

- `visit_For`. The for-loop shape (FOR_ITER, GET_ITER, iter
  protocol opcodes) lands at v0.7.8 — the v0.7.7/v0.7.8 split of
  the original v0.7.7 roadmap row. modFor stays on the legacy
  classifier for one more release.
- `else` clause on `while`. The v0.6 classifier rejected it too;
  later phase B generalises.
- `break` / `continue`. Same — v0.6 rejected; later phase B widens.
- Compare / BoolOp / IfExp conds in `while`. The Name-only surface
  matches v0.6's modWhile exactly. Generalises after function
  bodies are in place.
- Multi-stmt body. The v0.6 modWhile classifier didn't accept
  these either; later phase B generalises.
- gopapy parser changes.
