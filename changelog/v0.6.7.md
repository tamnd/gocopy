# v0.6.7 ŌĆö codegen module-level: no-op statements

Date: 2026-04-29
Spec: notes/Spec/1500/1516_gocopy_v067_codegen_noops.md
CPython parity: Python/codegen.c (3.14.0): module-level
`compiler_mod` for bodies of bare statements with no side effects.

The v0.6.7 release teaches the codegen package the next module-
level shape after the v0.6.6 empty-module foundation: bodies of
N >= 1 no-op statements (`pass`, `None`, `True`, `False`,
`Ellipsis`, numeric literals, byte literals). The classifier still
owns every other shape.

The 246/246 oracle stays byte-identical. The parallel-output gate
(`TestCodegenParity`) automatically lights up every fixture whose
body matches the new shape; today that adds ~17 fixtures to the
codegen path.

## Note on roadmap deviation

Spec 1508 originally allocated v0.6.7 to function and closure
codegen. v0.6.6 shipped as foundation only (one shape) so v0.6.7
keeps filling in module-level shapes. Function codegen lands once
the module-level surface is broad enough that the parity gate has
solid coverage. Spec slots from 1517 onward retain their roadmap
allocations.

## Added

- `compiler/codegen/visit_module.go` ŌĆö module-level emitters split
  out of `codegen.go`. Houses `buildEmptyModule` (moved from v0.6.6)
  and the new `buildNoOpsModule`. The latter emits
  `RESUME 0; (N-1)x NOP; LOAD_CONST 0; RETURN_VALUE` with each
  non-last NOP carrying its statement's Loc and the trailing
  `LOAD_CONST + RETURN_VALUE` pair sharing the last statement's
  Loc. The PEP 626 encoder collapses RESUME into the synthetic
  prologue, emits per-stmt entries, and merges the trailing pair ŌĆö
  byte-identical to `bytecode.LineTableNoOps`.

- `compiler/codegen/source.go` ŌĆö local copies of the source-text
  helpers (`splitLines`, `stripLineComment`, `trimRight`,
  `lineEndCol`) the no-op shape uses for end-of-line column
  calculation. The parser AST does not yet record end positions,
  so codegen reads them from the source bytes the same way the
  classifier does.

## Changed

- `compiler/codegen/codegen.go` ŌĆö `Options` grows a `Source []byte`
  field for end-column lookups. `Build` recognizes the no-op shape
  via `classifyNoOpModule` (Pass or non-string Constant ExprStmt
  for every statement). Anything else still returns
  `ErrUnsupported`.

- `compiler/compiler.go` ŌĆö passes `source` into `codegen.Options`.

- `compiler/codegen_parity_test.go` ŌĆö picks up the modNoOps
  classifier path so the parity gate validates every newly-codegen-
  owned fixture against the classifier reference.

## Tests

- 246/246 fixtures byte-identical (`tests/run.sh`).
- `compiler/codegen/codegen_test.go` covers single `pass`, three-
  statement no-op chains, and the unsupported-assign fallback case.
- `compiler/codegen_parity_test.go` automatically validates every
  fixture whose body is empty or pure no-ops; today that is
  ~18 fixtures.

## Out of scope

- Module-level docstring (modDocstring) ŌĆö next sub-release.
- Module-level assignments (modAssign and friends) ŌĆö later.
- Function definitions, closures ŌĆö later.
- Any optimizer pass ŌĆö v0.6.8 / v0.6.9.
