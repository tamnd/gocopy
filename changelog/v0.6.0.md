# v0.6.0 — core types: Loc, Atom, Const, OpMeta, oparg helpers (2026-04-28)

This is the first release in the v0.6.x line. It is the foundation
for the rest of the series: it lands a universal vocabulary the
later sub-releases will consume, and it cleans out the magic-number
pile in `compiler/` so future passes can be written against named
helpers.

The 246/246 oracle stays green. No behavior change. No wire-format
change. Pure additive new types plus a mechanical migration of the
17 magic-number sites.

## What changed

### New types in `bytecode/`

- **`bytecode/loc.go`** — `Loc{Line, EndLine uint32; Col, EndCol uint16}`,
  matching CPython's `code.co_positions()` 4-tuple shape. `Valid`,
  `IsZero`, `OneLine` predicates.

- **`bytecode/atom.go`** — `Atom uint32` plus `AtomTable` for
  deterministic string interning. Insertion order equals atom ID, so
  one table can drive `co_names` ordering.

- **`bytecode/const.go`** — `Const` tagged union covering every
  marshal-able co_consts value (None / True / False / Ellipsis /
  int / float / complex / str / bytes / tuple / code), plus
  `ConstPool` with type-specific interning that mirrors CPython:
  singletons for None/True/False/Ellipsis, content-keyed for
  str/bytes, value-keyed for int, bit-pattern-keyed for float and
  complex, ref-tuple-keyed for tuples, never-interned for code
  objects.

- **`bytecode/opmeta.go`** — `OpMeta{Name, HasArg, CacheSize,
  ArgKind, Flags, StackEff, StackVar}` plus a 256-entry table
  covering every opcode named in `bytecode/opcode.go`. Cross-checked
  against `python3.14`'s `opcode.opmap`, `opcode._inline_cache_entries`,
  `opcode.hasarg`, and `dis.stack_effect`.

- **`bytecode/oparg.go`** — typed helpers replacing the
  bit-fiddling that used to live inline:
  - `LoadGlobalArg(nameIdx, pushNull)` for `(idx<<1)|nullBit`.
  - `LoadAttrArg(nameIdx, asMethod)` for the same layout on LOAD_ATTR.
  - `LflblflbArg(slotL, slotR)` for the `(L<<4)|R` super-instruction.
  - `CompareCondArg(base)` for the `+16` conditional-context bump.
  - `LocalsKindLocal`, `LocalsKindArg`, `LocalsKindArgCell`,
    `LocalsKindFree`, `LocalsKindCell` named constants for
    `co_localspluskinds` bytes.

### Extended `bytecode/flags.go`

Added the five `co_flags` bits CPython 3.14 defines beyond what we
already had:

- `CO_HAS_DOCSTRING = 0x04000000`
- `CO_NO_MONITORING_EVENTS = 0x02000000`
- `CO_FUTURE_ANNOTATIONS = 0x01000000`
- `CO_FUTURE_GENERATOR_STOP = 0x00800000`
- `CO_FUTURE_BARRY_AS_BDFL = 0x00400000`

Plus the `co_localspluskinds` byte constants matching
`Include/cpython/code.h`: `FastHidden`, `FastLocal`, `FastCell`,
`FastFree`, `FastArgPos`, `FastArgKw`, `FastArgVar`, and the
shorthand `FastArg = FastArgPos | FastArgKw`.

### Magic-number migration

Seventeen call sites flipped from raw bytes to named helpers. No
byte changes; the oracle proves parity:

- `compiler/func_body.go` — fourteen sites (LFLBLFLB packs,
  COMPARE_OP `+16`, `CO_HAS_DOCSTRING`, LocalsPlusKinds bytes,
  LOAD_GLOBAL/LOAD_ATTR push-NULL bits).
- `compiler/compiler.go` — three sites (LocalsPlusKinds bytes for
  the simple-funcdef and closure shapes).
- `compiler/mixed.go` — one site (LocalsPlusKinds byte for
  modMixed funcdefs).

### Cross-check tests against CPython

`bytecode/cpython_parity_test.go` shells out to `python3.14` via
`os/exec` and validates every entry in `OpMetaTable` against
CPython's introspection: `opcode.opmap` for opcode numbers,
`opcode._inline_cache_entries` for cache sizes, `opcode.hasarg` for
the HasArg flag, and `dis.stack_effect` for fixed-effect opcodes.
The test calls `t.Skip` if `python3.14` is not on PATH, so the
basic CI test job (no Python) stays green; the oracle CI job has
Python and runs them.

## What did not change

- No marshal wire-format change.
- No CodeObject field changes; `Consts []any` stays as-is.
- No pipeline pass moved to the new types yet; this release is
  scoped to additive vocabulary plus the mechanical migration.

The full field-type migration (every `line int / col byte` flipped
to `Loc`) and the `co_consts []any` swap to `[]Const` are deferred
to v0.6.0.x or v0.6.1, after the foundation types prove themselves.
