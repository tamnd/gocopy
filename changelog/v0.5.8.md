# v0.5.8 — Name-default parameters in modMixed functions (2026-04-28)

## What changed

### Default parameters with Name expressions

Functions in modMixed modules can now have default parameter values where the
default is a module-level name (a Name expression, loaded via LOAD_NAME at
module level). For example, `def setraw(fd, when=ONE_THIRD): return fd` where
`ONE_THIRD` is a module-level constant.

CPython's pattern for K Name defaults is:

    LOAD_NAME default_0        # 1 CU each, at (defLine, defLine, colStart, colEnd)
    ...
    LOAD_NAME default_K-1
    BUILD_TUPLE K              # \
    LOAD_CONST <code object>   #  | LONG entry: 5 CU (non-last) or 7 CU (last)
    MAKE_FUNCTION              #  |
    SET_FUNCTION_ATTRIBUTE 1   #  |
    STORE_NAME funcName        # /

`SET_FUNCTION_ATTRIBUTE 1` sets the `__defaults__` attribute on the function
object. The function's code object itself is unaffected — co_argcount counts
all parameters including those with defaults.

### Changes to classification

`extractFuncBodyExpr` now accepts functions with `*parser2.Name` defaults.
Functions with any other default expression type (constants, calls, etc.) are
still rejected.

### Changes to linetable

`MultiFuncDefEntry` gains a `Defaults []DefaultInfo` field. Both
`MixedModuleLineTable` and `MultiFuncDefLineTable` emit one 1-CU entry per
default load before the funcdef LONG entry. The funcdef LONG entry uses 5 CU
(non-last with defaults) or 7 CU (last with defaults) instead of 3 or 5.

### Single-line docstring encoding fix

`MixedModuleLineTable` previously always used the LONG entry format for the
docstring, even for single-line docstrings. CPython uses the compact ONE_LINE1
encoding when the docstring fits on one line. Fixed by branching on
`DocEndLine == DocLine`.

## New fixture

- `243_default_param.py` — module docstring + ONE_THIRD folded assign +
  `rgb_to_yiq(r,g,b)` (no defaults) + `setraw(fd,when=ONE_THIRD)` (1 default)
